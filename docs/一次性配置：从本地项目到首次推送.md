# 一套“从零开始配置 + 排错 + 后续继续提交”的完整操作手册。

---

## 一次性配置：从本地项目到首次推送

### 1. 本地先确认这是一个 Git 仓库

在项目根目录执行：

```powershell
git status
```

如果能看到分支、文件状态，说明当前目录已经是 Git 仓库。
如果提示不是 Git 仓库，就先初始化：

```powershell
git init
```

---

### 2. 先把 `.gitignore` 配好

你这次的核心目标是：

* `config/` 以后不提交
* `workflows/` 以后不提交
* `.idea/`、`__pycache__/`、`*.pyc` 这种开发缓存也不提交

建议 `.gitignore` 至少写成这样：

```gitignore
config/
workflows/
.idea/
.venv/
.pytest_cache/
__pycache__/
*.pyc
```

如果 `.gitignore` 出现乱码、记事本打开是怪字符、VS Code 说是二进制，说明编码坏了。最稳的做法不是修，而是直接重建：

```powershell
@"
config/
workflows/
.idea/
.venv/
.pytest_cache/
__pycache__/
*.pyc
"@ | Set-Content .gitignore -Encoding ascii
```

检查内容：

```powershell
Get-Content .gitignore
```

验证忽略规则是否真的生效：

```powershell
git check-ignore -v config/model_resources.json
git check-ignore -v workflows/111/workflow.yaml
```

只要能显示命中的 `.gitignore` 规则，就说明忽略成功。

---

### 3. 只写 `.gitignore` 还不够：已经被跟踪过的文件要“取消跟踪”

这是很多人第一次会踩的坑。

`.gitignore` 只对“未被跟踪的新文件”生效。
如果某个目录以前已经被 Git 管过，光写进 `.gitignore` 没用，必须把它从索引里移出去。

你的场景里，典型就是：

* `config/`
* `workflows/`
* `.idea/`
* `api/__pycache__/`
* `core/__pycache__/`

做法：

```powershell
git rm -r --cached config workflows .idea api/__pycache__ core/__pycache__
```

这条命令的含义是：

* 从 Git 跟踪中移除
* 但本地文件仍然保留
* 以后因为 `.gitignore` 存在，Git 不会再继续管它们

---

### 4. 创建远程 GitHub 仓库

在 GitHub 上新建仓库时，建议这样选：

* `Add README`：关掉
* `Add .gitignore`：No .gitignore
* `Add license`：No license

也就是创建一个空仓库。
原因很简单：你本地已经有项目了，远程也再自动加文件的话，第一次推送更容易冲突。

---

### 5. 绑定远程仓库

创建完 GitHub 仓库后，复制 HTTPS 地址。
然后在本地执行：

```powershell
git remote add origin 你的仓库地址
git remote -v
```

如果 `git remote -v` 能看到 `origin` 的 fetch 和 push 地址，说明绑定成功。

如果以后要改远程地址，用：

```powershell
git remote set-url origin 新地址
```

---

### 6. 首次提交与推送

先检查状态：

```powershell
git status
```

然后按下面的方式收口：

```powershell
git add .gitignore
git add -u
git add 你想提交的新文件路径
git status
git commit -m "可视化运行进程"
git push -u origin master
```

这里有一个很重要的习惯：

* `git add -u`：只收“已被跟踪文件”的修改和删除
* `git add 文件路径`：显式加入你想提交的新文件
* 不要一上来就无脑 `git add .`

这是你这次能避开很多脏文件的关键。

## 这次你遇到过的典型问题，分别怎么判断和修复

### 1. `fatal: 'origin' does not appear to be a git repository`

意思是：本地没有配置 `origin` 这个远程仓库，或者配置错了。

先查：

```powershell
git remote -v
```

如果为空，说明还没加远程：

```powershell
git remote add origin 仓库地址
```

---

### 2. `info: please complete authentication in your browser...`

这个不是报错，是正常现象。
意思是 GitHub 让你去浏览器完成一次登录认证。认证完成后，推送就会继续。

---

### 3. 明明 `git push` 成功了，GitHub 仓库看起来却没文件

这次你遇到的根因是：

* 你推送到了 `master`
* GitHub 默认打开的是 `main`

所以看起来像“仓库没内容”，其实代码在另一个分支。

解决方式有两个：

第一种，直接切到 `master` 看。
第二种，把 GitHub 默认分支改成 `master`。

以后如果你想统一成 `main`，可以再单独改分支名。

---

### 4. `.gitignore` 乱码、像二进制文件

说明编码损坏了。直接重建，不要修旧文件：

```powershell
@"
config/
workflows/
.idea/
.venv/
.pytest_cache/
__pycache__/
*.pyc
"@ | Set-Content .gitignore -Encoding ascii
```

---

### 5. `.gitignore` 明明写了，为什么 `config/`、`workflows/` 还在 `git status` 里

因为它们之前已经被 Git 跟踪过。
这时必须：

```powershell
git rm -r --cached config workflows
```

记住一句话：

`.gitignore` 负责“以后别再跟踪”
`git rm --cached` 负责“把已经跟踪的移出去”

两者缺一不可。

---

### 6. `git status` 同时出现 `D` 和 `??`

例如：

* `D workflows/...`
* `?? workflows/...`

这通常不是异常，而是这个过程中的过渡状态：

* `D`：索引里已删除，说明这次提交会把它从仓库移除
* `??`：本地文件仍然在，但因为忽略规则还没完全生效，Git 把它当未跟踪文件又看见了

修正方法就是先把 `.gitignore` 配对，再重新看状态。

---

### 7. 根目录出现了一个奇怪文件：`git add .`

这是误操作产生的异常文件名。
判断它是不是存在，可以用：

```powershell
Get-ChildItem -Force | Where-Object { $_.Name -eq 'git add .' }
```

这种带尾部点的 Windows 文件，普通删除命令有时会失效。
如果后面你还遇到这种怪文件，优先先确认它是不是文件对象，再删。必要时用 `cmd` 的原始路径删除。

但正常情况下，只要以后别把命令误写进重定向或文件创建流程里，就不会再出现。

## 以后继续提交，标准流程怎么走

以后你每次开发完，建议固定按这个顺序走。

### 情况 A：你只改了已有文件，没有新建文件

```powershell
git status
git add -u
git status
git commit -m "你的提交说明"
git push
```

这是最稳的常规流程。

---

### 情况 B：你既修改了旧文件，又新建了几个明确要提交的新文件

```powershell
git status
git add -u
git add 新文件1 新文件2
git status
git commit -m "你的提交说明"
git push
```

也就是：

* 旧文件修改：`git add -u`
* 新文件：显式点名加进去

---

### 情况 C：你不确定有没有脏文件，不想误提交

先看状态：

```powershell
git status
```

如果看到你不想提交的内容：

* `config/`
* `workflows/`
* `.idea/`
* `__pycache__/`
* `.pyc`

先停下来，不要直接 `git add .`

先检查：

```powershell
git check-ignore -v 具体文件路径
```

如果没命中，就说明 `.gitignore` 还需要补规则。

---

## 一个很实用的提交前检查模板

以后每次提交前，直接按这个小模板走：

```powershell
git status
git add -u
git add 需要提交的新文件路径
git status
git commit -m "本次提交说明"
git push
```

你重点只看第二次 `git status`：

如果只剩你想提交的代码改动，就可以 commit。
如果出现奇怪文件，就先停，不要硬推。

## 如果你想把默认分支统一成 `main`

你现在已经成功推到了 `master`。这没有问题。
但如果你想和 GitHub 默认习惯统一，可以后面再做这件事：

```powershell
git branch -m master main
git push -u origin main
```

然后去 GitHub 仓库设置里把默认分支改成 `main`。
等确认没问题后，再删除远程旧的 `master`。

这一步不是必须。你现在维持 `master` 也完全可以。
