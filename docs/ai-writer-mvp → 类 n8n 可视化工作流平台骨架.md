# ai-writer-mvp → 类 n8n 可视化工作流平台骨架

## 后续工作目标与执行顺序说明

当前项目已经从“文章生成器”推进到一个**可运行的可视化工作流编辑器 MVP**。
它已经具备：

* React Flow 可视化编辑
* `input / prompt / output` 三类节点
* 节点增删改
* 前后端统一接口
* editor schema 与 YAML schema 互转
* 基于拓扑排序的工作流执行
* Run Inputs / Run Result / steps 展示
* 节点执行高亮
* 基础保存前校验

这说明项目已经完成了“最小闭环”。
接下来，不应该无序加功能，而要按**平台化演进**来推进。

---

# 一、当前项目所处阶段

当前项目更准确的定位是：

> **AI 工作流编辑器 MVP**

还不是完整的 n8n 类平台，原因在于当前仍然偏向：

* 单工作流编辑
* AI prompt 链式生成
* 三类基础节点
* 手动运行
* 基础图结构和基础状态流转

离“平台骨架”还差：

* 更丰富的节点能力
* 更强的节点配置能力
* 更完整的工作流管理能力
* 更清晰的数据流 / 控制流设计
* 更强的 schema 扩展能力
* 更适合重构和演进的代码分层

所以后续重点不是“想到什么加什么”，而是：

> **先把 schema 与节点配置能力做厚，再扩展节点类型，再考虑平台级能力。**

---

# 二、后续总目标

后续目标可以分成三层：

## 第一层目标：把当前编辑器做稳

目标是让现有系统从“能跑”提升到“可持续开发”。

包括：

* 配置更严谨
* 错误提示更明确
* schema 更可扩展
* 节点能力更统一
* 前后端结构更容易演进

这一层是最优先的。

---

## 第二层目标：把 AI 工作流能力做厚

目标是让项目从“固定文章链”变成“可扩展 AI 工作流”。

包括：

* prompt 节点支持更强配置
* 节点内直接编辑 prompt
* 模型参数可配置
* 增加文本处理与判断类节点
* 让流程表达力更强

这一层是项目从 MVP 走向产品雏形的关键。

---

## 第三层目标：向平台骨架推进

目标是逐步接近 n8n 类平台。

包括：

* 多工作流管理
* 执行历史
* Trigger / Webhook
* 条件 / 分支 / 合并
* 端口化连接
* 插件化节点

这一层不能现在一口气做完，要在前两层稳定后逐步推进。

---

# 三、开发顺序总原则

后续所有功能，都建议按下面原则排序：

## 1. 先增强 schema，再增强 UI

因为前端页面只是表象，真正决定后面能不能扩展的是：

* 节点配置结构
* YAML 存储结构
* engine 执行结构
* 前后端互转逻辑

所以先把 schema 想清楚，再做面板和交互。

---

## 2. 先增强 prompt 节点，再增加新节点

因为你当前项目本质仍然是 AI Writer 演进而来，最核心节点就是 prompt 节点。

所以优先级应该是：

* prompt 节点能力增强
* 再考虑 condition / transform / http 等新节点

而不是一上来猛加很多节点类型。

---

## 3. 先做低风险高收益功能

优先做：

* 对现有结构改动小
* 对未来扩展帮助大
* 能明显提高开发效率和项目稳定性

的功能。

---

## 4. 到达结构拐点时，允许停下来重构

后续有些阶段不能只靠“补丁式开发”，而需要停下来重构。

尤其是出现这些情况时：

* `NodeConfig` 字段越来越臃肿
* `WorkflowEditor.jsx` 体积越来越大
* `engine.py` 的 `run_node()` 分支越来越复杂
* `api/workflows.py` 的 schema 互转越来越难维护
* 新增节点时要改很多地方且容易漏改

一旦出现这些现象，就应该停下来做一次小规模结构重构。

### 优先级最高，先做完这一批

---

## 1. 前端保存失败时显示后端 detail

### 目标

现在保存失败如果只提示 `Save failed`，调试成本高。
应直接展示后端返回的 `detail`。

### 价值

* 成本低
* 收益立刻可见
* 后续调试所有新功能都更省力

### 改动范围

* `frontend-react/src/WorkflowEditor.jsx`

### 难度

低

### 优先级

最高，建议立刻做

---

## 2. Prompt 节点支持“模板模式 / 内联模式”

### 目标

现在 prompt 节点只能选择 prompt 名称，例如：

* outline
* draft
* refine

后续应支持两种模式：

* `template`：选择已有 prompt 文件
* `inline`：直接在节点里编辑 prompt 文本

### 价值

这是最适合你当前项目定位的增强功能。
它能明显提高“像平台”的感觉，同时不会立刻把架构搞乱。

### 为什么这一步很关键

因为这一步会推动你升级节点 schema，但又不会一下子把项目打散。

### 建议新增字段

可以考虑给 prompt 节点增加：

* `promptMode`
* `prompt`
* `promptText`

例如：

```yaml
promptMode: template
prompt: outline
promptText: ""
```

或者：

```yaml
promptMode: inline
prompt: ""
promptText: "请根据{topic}生成文章大纲"
```

### 涉及文件

* `frontend-react/src/NodeConfigPanel.jsx`
* `frontend-react/src/WorkflowEditor.jsx`
* `api/workflows.py`
* `core/engine.py`
* YAML 示例文件

### 难度

低到中

### 优先级

非常高，建议作为下一项核心功能

---

## 3. 把 prompt 节点的 LLM 配置正式接入编辑器

### 目标

你当前 `engine.py` 已经支持：

* model
* temperature
* timeout
* max_retries

但前端和 schema 还没有把它正式纳入，导致 YAML 里的 `llm` 配置可能在保存时丢失。

### 价值

这一步非常重要，因为它把“模型配置权”交给节点，而不是写死在后端默认值里。

### 建议字段

给 prompt 节点加入：

* `llm.model`
* `llm.temperature`
* `llm.timeout`
* `llm.max_retries`

### 涉及文件

* `NodeConfigPanel.jsx`
* `WorkflowEditor.jsx`
* `api/workflows.py`
* `core/engine.py`

### 难度

中

### 优先级

高，建议紧跟 prompt 内联编辑之后

# 五、从重构角度看，哪些阶段应该停下来整理

不是所有阶段都应该一路加下去。
有些节点功能加到一定程度，必须先停下来整理。

---

## 应该第一次停下来重构的时机

当你完成下面 3 项之后：

* prompt 内联编辑
* LLM 参数编辑
* 工作流切换

就应该停下来做一次**第一次结构整理**。

### 这次整理重点

#### 1. 抽离前端 schema 工具函数

把 `WorkflowEditor.jsx` 里的一些配置构造逻辑、节点默认值逻辑、节点转换逻辑抽出去。

可考虑新建：

* `frontend-react/src/workflowSchema.js`
* `frontend-react/src/nodeDefaults.js`

#### 2. 拆分 `WorkflowEditor.jsx`

当前它已经比较长了，后面一定会继续变大。
建议拆出：

* `RunPanel.jsx`
* `RunResultPanel.jsx`
* `WorkflowToolbar.jsx`

#### 3. 后端 schema 转换函数单独整理

`api/workflows.py` 现在同时做：

* 接口
* Pydantic schema
* YAML 互转
* 校验

后面会越来越重。
建议后续拆成：

* `api/schemas.py`
* `core/workflow_schema.py`

---

## 应该第二次停下来重构的时机

当你准备引入：

* 条件节点
* 文本处理节点
* 更多节点类型

时，建议停下来重构 `engine.py`。

### 当前 `engine.py` 的问题

现在执行逻辑是：

```python
if node_type == "input":
    ...
if node_type == "prompt":
    ...
if node_type == "output":
    ...
```

节点类型一多，这里会越来越难维护。

### 后续建议方向

改成注册式或分发式：

* `run_input_node`
* `run_prompt_node`
* `run_output_node`
* `run_condition_node`
* `run_transform_node`

再进一步可以做成：

```python
NODE_RUNNERS = {
    "input": ...,
    "prompt": ...,
    "output": ...
}
```

这样后面新增节点时，不用一直在主逻辑里堆 `if/elif`。
