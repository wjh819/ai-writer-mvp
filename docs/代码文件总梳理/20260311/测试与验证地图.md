# 测试与验证地图

## 1. 文档目的

这份文档解决的不是“代码怎么组织”，而是：

> 当你改完某类功能后，应该跑哪些测试、做哪些手动验证、按什么顺序验证，才能更快判断这次改动是否真的生效。

它主要回答五个问题：

1. 当前项目里，哪些行为已有自动化覆盖
2. 哪些行为仍主要依赖手测
3. 改某类文件后，最小应该跑哪些验证
4. 常规启动命令分别在验证什么
5. 如果只知道文件名，应该反查哪条验证路径

这份文档适合作为：

* 改动完成后的验证速查表
* 二次开发时的回归参考
* 重构后判断“改坏没改坏”的辅助文件

---

## 2. 当前验证体系总览

当前项目的验证体系可以分成六层。

### 2.1 前端纯逻辑自动化测试

主要覆盖 `frontend-react/src/workflow-editor/` 下已经拆分出的纯逻辑层，包括：

* 图同步规则
* 节点工厂
* 映射层
* 输出规则校验
* 保存前组合校验
* 运行输入派生
* 选择状态派生
* 执行态视图派生

这一层的特点是：

* 跑得快
* 不依赖真实页面
* 是前端当前最稳定的自动化主力

---

### 2.2 前端入口 / 轻量挂载 smoke test

主要覆盖：

* `App.jsx` 是否还能挂载主页面
* `WorkflowEditor.jsx` 在降载 mock 条件下能否显示关键控件

这一层的作用不是验证完整交互，而是防止出现：

* 页面白屏
* 组件路径断裂
* 主入口装配失效

---

### 2.3 后端规则层 pytest

当前主要覆盖：

* `api/workflow_converter.py`
* `api/workflow_validator.py`

这一层主要验证：

* 编辑器 schema 与 YAML 的转换
* 保存前后端合法性校验规则

---

### 2.4 页面手测

当前前端有一部分关键区域仍主要依赖页面手测，例如：

* ReactFlow 真正的连边 / 选中 / 删除行为
* 右侧配置面板表单交互
* Save / Run 按钮在真实页面中的行为
* 运行结果面板显示效果
* workflow 切换与刷新

---

### 2.5 接口 / 联调验证

主要通过：

```bash
uvicorn fastapi_app:app --reload
cd frontend-react
npm run dev
```

来验证：

* FastAPI 路由是否正常
* 前后端接口是否能联通
* save / load / run / prompts 是否能在真实页面工作

---

### 2.6 引擎直跑与结构检查

主要通过：

```bash
streamlit run app.py
python visualize.py workflows/article.yaml
```

分别验证：

* 引擎运行链路
* workflow 结构可视化是否正常

这两条路径属于补充验证，不替代前端测试或后端测试。

#### 运行结果展示组件无独立测试

## 4. 后端测试总览

当前后端自动化测试主要集中在“规则层”，不是完整运行层。

### 4.1 已有自动化覆盖的后端区域

#### 转换层

对应文件：

* `api/workflow_converter.py`
* `tests/test_workflow_converter.py`

已覆盖行为：

* `build_default_output`
* YAML → editor schema
* editor schema → YAML
* output fallback
* llm 字段转换
* inputs / edges / position 转换
* round-trip 稳定性

#### 校验层

对应文件：

* `api/workflow_validator.py`
* `tests/test_workflow_validator.py`

已覆盖行为：

* 空 workflow
* node id 非空 / 唯一
* node type 合法
* output 非空 / 前缀 / 格式 / 唯一性
* output / node id 冲突
* prompt 节点 template / inline 规则
* output 节点 input 要求
* edge source / target 存在性
* edge-input consistency

---

### 4.2 当前后端自动化覆盖缺失的区域

以下文件当前未见自动化测试：

* `api/workflows.py`
* `fastapi_app.py`
* `core/engine.py`
* `core/llm.py`
* `utils/prompt_loader.py`
* `app.py`
* `visualize.py`

这意味着下面这些能力目前不能算“已有自动化覆盖”：

* FastAPI 路由装配是否正确
* `/workflows` / `/run` / `/prompts` 是否完整正常
* engine 拓扑执行链路
* prompt 文件加载
* llm 外部调用
* Streamlit 页面逻辑
* visualize 脚本逻辑

## 8. 推荐验证顺序

### 1）先跑最贴近改动点的自动化测试

例如改 graph，就先跑 graph test。
改 validator，就先跑 validator test。

这样可以最快定位问题是不是出在你刚改的那一层。

### 2）再跑该侧的全量测试

前端：

```bash
cd frontend-react
npm test
```

后端：

```bash
pytest
```

这样可以确认改动有没有把同侧别的逻辑带坏。

### 3）再启动接口层

```bash
uvicorn fastapi_app:app --reload
```

并查看：

```text
http://127.0.0.1:8000/docs
```

这样可以确认后端不是“测试过了，但实际起不来”。

```bash
cd frontend-react
npm run dev
```

联调时重点看：

* workflow list
* load
* save
* prompts
* run
* 页面是否白屏
* 控件是否可点

### 5）如果改动涉及运行链路，再跑 Streamlit

```bash
streamlit run app.py
```

适用情况：

* engine
* llm
* prompt
* workflow
* run 流程

### 6）如果改动涉及 workflow 结构，再补 visualize

```bash
python visualize.py workflows/article.yaml
```

适用情况：

* 节点 / 边结构改动
* workflow 文件结构改动
