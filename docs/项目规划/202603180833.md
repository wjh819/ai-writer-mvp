一，编辑过程允许临时未完成，但关键键和值不得进入明显非法状态

已经符合的部分：

保存前你已经有一套比较完整的底线校验。
包括：
output 格式、
output 唯一性、
output 与 node id 冲突、
prompt 节点必要字段、
output 节点至少一个输入、
edge 与 target.inputs 一致性。

这说明“最终落盘前守住底线”这件事已经成立。

明显不符合或存在风险的部分：

当前还不能证明关键字段在编辑过程中被及时拦住。
尤其是：
node id、
output。

你虽然有 `validateSingleNodeOutput()`，
但我没看到它已经接到真实编辑链路上。
所以现在更像是“保存前拦”，而不是“关键键值编辑时就尽量不进入明显非法状态”。

另外，renameNode 这条链虽然会拦空值和重名，但它的错误仍在 controller 里直接 alert，这和你第 4 条规则冲突，后面要一起收口。

当前结论：

部分符合，但没有完全落实。
保存前底线成立；
编辑态对核心键值的即时防线还没闭合。

二，所有图结构变更必须通过统一 graph action 链落盘

已经符合的部分：

你现在大部分关键图操作已经走 action / graph 链路了：
新增节点、
更新节点、
重命名节点、
删除节点、
复制节点、
连边、
删边。

这说明主干方向已经成立。

明显不符合的部分：

`onNodesChange` 还没有看到同等级统一 action 入口。
也就是说，节点变化并没有像边变化那样被完整收口到统一 graph action 链。

这是当前最直接的不符合项之一。
因为你这条规则已经说得很绝对了：“所有图结构变更必须通过统一 graph action 链落盘。”
那目前这条在实现上还没有完全兑现。

当前结论：

主干大体符合，但未完全符合。
这是当前最需要补的规则实现缺口之一。

三，output 属于核心领域键，禁止按普通展示字段对待

已经符合的部分：

这一点其实你的代码已经很接近了，而且很多地方已经在按核心键处理。

体现包括：
runInputs key 依赖 output；
graph contract 依赖 source.output 写入 target.inputs；
rename 时 output 有专门跟随规则；
复制节点时 output 要重新生成唯一值；
validators 对 output 做格式、唯一性、冲突校验；
helpers 里专门有 output 相关的领域函数。

这些都说明 output 在实际实现里已经不是普通展示字段。

仍有风险或不够彻底的部分：

手动修改 output 的编辑链路，当前还没有完全证明会被“领域级特殊处理”到底。
也就是说：
虽然你整体上把它当核心键了，
但在具体编辑入口上，还没有完整证据证明每次修改都会走足够强的规则保护。

当前结论：

基本符合，而且是目前落实程度比较高的一条；
但编辑入口上的防线还不够完整。

四，底层只返回结果，所有用户可见错误由页面消费层统一决定

已经符合的部分：

runtime 这条线已经明显在往这个方向走了。
比如 Save / Run / Refresh 大体上都是返回结果对象，由 `WorkflowEditor.tsx` 决定 alert 或后续提示方式。

这是正确方向。

明显不符合的部分：

`useWorkflowGraphEditor.renameNode()` 里仍然直接 `alert(result.error)`。

这条是明确不符合。
而且不是风格问题，是规则层面的不一致。

如果这条不收口，后面你一旦扩展为：
toast、表单级错误、顶部提示、字段内联错误，
graph 相关操作会成为一块例外区，维护会越来越别扭。

当前结论：

方向符合，但实现未统一。
这是当前最明确的架构残留之一。

五，session 作为独立业务线与普通 runtime 并行演进，只共享底层能力

当前结论：

基本符合，且暂时没有看到明显结构性违背。

总表结论

如果把这 5 条规则放到你当前代码上，结论是：
