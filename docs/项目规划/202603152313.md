

先给结论版顺序：

1. session / run 展示层收口
2. `SessionRunDetail.steps` 前后端口径收紧
3. run 详情增强与调试信息展示
4. session 生命周期管理
5. workflow run 与 session run 公共展示抽象
6. run 对比能力
7. 节点级 conversation 语义设计与最小字段落地
8. conversation 运行时继承链打通（新对话 / 继承指定节点）
9. conversation 展示与调试能力
10. 简单新节点类型
11. 中等复杂节点类型
12. 执行模型级扩展（条件、并行、循环、子流程）

再讲一下为什么我把 conversation 能力放在第 7、8、9，而不是更靠前或更靠后。

它不能太靠前，是因为你现在虽然主链已成形，但 runtime/session 的展示层、`steps` 口径、run detail 解释层还在继续收口。conversation 一旦引入，本质上就是新增一种运行态对象。那时你不再只有：

* `state`
* `steps`
* `runResult`
* `latest_state`
* `SessionRunDetail`

还会多出一条新的“conversation 上下文链”。如果前面的运行态基础还不稳，conversation 会让系统可解释性进一步下降。

它也不应该太靠后。因为 conversation 继承能力，本质上是 LLM workflow 很基础的一层能力，比新增很多花哨节点更底层。你后面如果要做更像 agent 的节点、复杂 prompt 节点、多轮追问、基于历史对话推理，这个能力迟早都要补。而且它一旦设计得太晚，后面可能要返工 prompt 节点 schema、engine prompt 执行入口、run detail 展示层。

所以最合适的位置，就是放在：

基础 runtime/session/run 层基本收口之后，
大量新增节点类型和执行模型级扩展之前。

如果换成更口语化的路线，就是：

第一步，先把“现在已经跑起来了，但展示和解释还不够稳”的地方补齐。
也就是先做 session/run 展示层收口，加上 `steps` 口径统一、run detail 解释增强。这一步不炫，但会让后面所有开发都更稳。

第二步，把“开发者看系统”的能力做强。
包括 run detail、调试信息、失败原因、step 展示、state 展示、session 生命周期管理。这样你自己后面改系统会顺很多。

第三步，再做“公共抽象”。
等 workflow run 和 session run 的展示语义稳定后，再去抽公共展示组件或统一运行态展示逻辑。再把 run 对比做出来，形成更强的观察和回归能力。

第四步，再引入“conversation 作为正式能力层”。
先做概念和字段，不急着一口气做全。先支持：

* 默认新对话
* 继承指定节点对话
  这一步最关键的是把“数据输入来源”和“对话上下文来源”分开。

第五步，再把 conversation 真正打通到 engine 和展示层。
也就是让节点在运行时真的能：

* 用 A 的输出作为本轮 user message
* 继承 B 的 conversation history
* 并把自己的 conversation 结果留给后续节点使用
  然后补展示与调试能力，让这条链可解释。

第六步，再开始扩节点能力。
先从简单节点做起，不改变 engine 主执行模型；等这批做顺了，再考虑更复杂节点。

第七步，最后才做会改变执行模型的能力。
比如条件、循环、并行、子流程。这些不是不能做，而是太早做成本很高，也最容易把前面还没完全稳住的 runtime / conversation 语义放大。

如果单独只看“conversation 需求本身”该怎么拆，我建议你把它放进这条大路线里的三个子阶段。

conversation 第一阶段：语义与字段阶段。
目标是定义：

* prompt 节点默认新对话
* 可继承指定节点 conversation
* 文本输入来源与 conversation 来源分离
  这个阶段主要改 schema / types / validator / 前端节点配置，不急着做复杂 UI。

conversation 第二阶段：运行时打通阶段。
目标是让 engine 在执行 prompt 节点时，显式决定：

* 本轮是否新对话
* 如果继承，从哪拿 history
* 当前 user message 是什么
* 调用后如何产出新的 conversation 结果
  到这里为止，你的例子就能成立：A 提供文本，B 提供对话上下文，C 执行续聊。

conversation 第三阶段：展示与调试阶段。
目标是让 run detail / step detail 可见：

* 当前节点是新对话还是续聊
* 继承自哪个节点
* 本轮发送了什么
* 得到了什么结果
  这一层对长期可维护性非常关键。
